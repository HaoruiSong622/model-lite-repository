## 1. 引言

### 1.1 目的

本文档为 ModelLite 平台模型仓库（Repository）模块的需求规格说明书，旨在清晰、完整地描述该模块的功能需求、非功能需求、接口需求及约束条件，为后续的架构设计、特性设计和开发实现提供唯一依据。

### 1.2 适用范围

本文档涵盖 ModelLite 平台模型仓库模块的全部功能需求，不包含具体的技术实现方案或架构设计。

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 模型（Model） | 模型仓库中的顶层实体，代表一个机器学习模型的元数据集合 |
| 模型版本（Model Version） | 模型的子实体，代表模型某一特定版本的权重文件集合及其路径信息 |
| 纳管（Register） | 将外部已有的可访问存储路径（PVC或NFS）登记到模型仓库管理数据库中，作为模型版本的权重来源，不执行文件复制 |
| 上传（Upload） | 根据给定的外部存储路径（NFS/CIFS/PVC），将权重文件从原路径拷贝到平台内置的 PVC 下，作为模型版本的权重来源 |
| 资源组（Resource Group） | 平台的分权分域单元，模型权重等资源从属于某一资源组 |
| 模型分类（Model Classification） | 模型的一级分类，如 TextGeneration、ImageTextToText 等 |
| 模型类型（Model Type） | 模型的二级分类，从属于某一模型分类，如 glm-5、Qwen2.5-VL-7B 等 |
| 机机接口（M2M API） | 供平台其他模块（如模型推理、模型训练）调用的接口 |
| 人机接口（User API） | 供前端页面或用户直接调用的接口 |
| 版本锁（Version Lock） | 标记模型版本正被训推任务占用，锁定期间禁止删除该版本 |

## 2. 总体描述

### 2.1 系统上下文

ModelLite 平台部署于 Kubernetes 集群，包含以下核心模块：
- **Gateway**：流量入口，负责请求路由和认证
- **模型仓库（Repository）**：模型存储、版本管理、权重文件管理
- **模型推理（Inference）**：推理服务部署与管理
- **模型训练（Training）**：训练任务管理
- **模型评测（Evaluation）**：模型评测任务管理
- **Jupyter 模型开发**：交互式开发环境
- **Controller Manager**：CRD 生命周期管理

模型仓库作为平台的模型资源中心，为下游的推理部署、微调训练、评测任务等模块提供模型数据支撑。

### 2.2 模块定位

模型仓库模块是 ModelLite 平台的核心组件之一，负责：
- 机器学习模型的存储与版本管理
- 权重文件的上传与纳管
- 模型分类体系的维护
- 为下游模块提供模型数据查询与引用能力

### 2.3 用户角色

| 角色 | 描述 |
|------|------|
| 平台管理员（admin） | admin_group 中的唯一账户，可访问所有资源组的资源 |
| 普通用户 | 从属于某一普通资源组，可访问本资源组和 public 资源组的资源 |
| 系统模块 | 平台其他模块（推理、训练等），通过机机接口调用仓库服务 |

### 2.4 约束与假设

- 模型仓库模块运行于 Kubernetes 集群内
- 平台已具备统一的身份认证和 RBAC 鉴权体系
- 平台已具备统一的操作日志/审计服务
- 模型权重文件存储于平台统一管理的 PVC 或外部 NFS 中
- 模型分类和类型存在系统预设值

## 3. 功能需求

### 3.1 模型权重生命周期管理

#### 3.1.1 模型创建

**需求编号**: REQ-MODEL-001

**需求描述**: 系统应支持创建模型实体。创建模型时必须同时创建该模型的第一个版本，不允许创建空模型（即无版本的模型）。

**输入**: 模型元数据（名称、描述、模型分类、模型类型、标签、资源组等）及第一个版本的权重来源信息

**约束条件**:
- 同一模型分类和模型类型组合下，模型名称必须唯一
- 模型名称在创建后不可修改
- 模型创建后，资源组不可修改（设计上需保留后续支持修改的可扩展性）
- 模型在后台需维护全局唯一标识符（Model ID）

**前置条件**: 用户已登录，所属资源组已确定

**后置条件**: 模型实体及第一个版本被持久化，操作日志被记录

#### 3.1.2 模型查看

**需求编号**: REQ-MODEL-002

**需求描述**: 系统应支持查看模型的元数据信息及其下的所有版本列表。

**输入**: 模型 ID

**输出**: 模型元数据（名称、描述、分类、类型、标签、资源组、创建者、创建时间、更新时间等）及版本列表

**约束条件**:
- 版本列表默认按版本号降序排列（最新版本在前）
- 仅返回当前用户有权限查看的模型信息（受资源组可见性控制）

#### 3.1.3 模型修改

**需求编号**: REQ-MODEL-003

**需求描述**: 系统应支持修改模型的部分元数据。

**可修改属性**: 描述、模型分类、模型类型、标签

**不可修改属性**: 模型名称、资源组（资源组暂不开放修改，但设计需保留可扩展性）

**约束条件**:
- 修改模型分类/类型后，需保证新分类+类型组合下名称唯一性约束不被违反
- 仅模型创建者或具有相应操作权限的用户可修改

#### 3.1.4 模型版本创建

**需求编号**: REQ-VERSION-001

**需求描述**: 系统应支持为已有模型创建新版本。

**输入**: 模型 ID、版本权重来源信息

**约束条件**:
- 版本号采用自动递增整数规则（V1, V2, V3...），由系统自动分配，用户不可指定
- 版本号不允许跳号
- 创建新版本时，权重来源可选择「纳管」或「上传」方式

#### 3.1.5 模型版本查看

**需求编号**: REQ-VERSION-002

**需求描述**: 系统应支持查看模型版本的详细信息。

**输出**: 版本号、版本状态、权重存储路径（PVC路径或NFS路径）、权重类型（FP16、w8a8等，如已识别）、是否为纳管版本、创建时间、更新时间、校验状态等

#### 3.1.6 模型软删除

**需求编号**: REQ-DELETE-001

**需求描述**: 系统应支持对模型或模型版本进行软删除。软删除后的模型/版本进入回收站，可被恢复。

**删除粒度**:
- 模型级删除：删除模型时级联删除其下所有版本
- 版本级删除：可独立删除某一特定版本，不影响模型及其他版本

**约束条件**:
- 删除模型或版本前，系统必须检查该版本是否被锁定（即正在被推理/训练任务占用）
- 如版本处于锁定状态，禁止删除，并提示用户该版本正被其他任务占用
- 仅当版本未锁定时，才允许执行删除操作
- 软删除操作需记录操作日志
- 仅具有相应操作权限的用户可执行删除

#### 3.1.7 模型硬删除

**需求编号**: REQ-DELETE-002

**需求描述**: 系统应支持从回收站中对模型/版本进行硬删除。硬删除后模型/版本被彻底移除，不可恢复。

**约束条件**:
- 硬删除时，除删除数据库中的元数据记录外，必须同时删除存储上的实际权重文件
- 硬删除前必须检查版本是否被锁定，如被锁定则禁止删除
- 硬删除操作需记录操作日志
- 仅具有相应操作权限的用户可执行硬删除

#### 3.1.8 模型恢复

**需求编号**: REQ-DELETE-003

**需求描述**: 系统应支持从回收站中恢复被软删除的模型或模型版本。

**约束条件**:
- 恢复后的模型/版本恢复到软删除前的状态
- 恢复操作需记录操作日志
- 仅具有相应操作权限的用户可执行恢复

#### 3.1.9 回收站管理

**需求编号**: REQ-RECYCLE-001

**需求描述**: 系统应提供回收站功能，展示所有被软删除的模型和版本。

**约束条件**:
- 回收站为全局统一列表
- 回收站中的模型/版本可见性受资源组控制：普通用户只能看到本资源组和 public 资源组中被删除的模型/版本；admin 可看到所有资源组中被删除的模型/版本
- 回收站支持查看、恢复、硬删除操作

#### 3.1.10 模型列表查询

**需求编号**: REQ-QUERY-001

**需求描述**: 系统应支持模型列表的分页查询、筛选和排序。

**查询能力**:
- 分页：支持分页浏览
- 筛选：支持按模型分类、模型类型、标签、模型名称等条件筛选
- 排序：支持按创建时间、更新时间等排序
- 搜索：支持全局搜索（跨分类搜索模型名称）

**约束条件**:
- 仅返回当前用户有权限查看的模型（受资源组可见性控制）

### 3.2 权重纳管

#### 3.2.1 纳管方式

**需求编号**: REQ-REGISTER-001

**需求描述**: 系统应支持通过给定存储路径对已有模型权重进行纳管。

**纳管定义**: 纳管是指获取外部已有的可访问存储路径（PVC或NFS），并将其注册到模型仓库管理数据库中，作为模型版本的权重来源。纳管过程不执行文件复制操作。

**纳管特性**:
- 系统仅记录用户提供的存储路径作为模型版本的权重来源
- 版本权重路径指向用户提供的原始路径
- 纳管过来的权重版本只能以只读方式挂载，禁止写入
- 查询接口中需返回该模型版本是否为纳管类型

**输入**: 模型 ID、存储路径信息（PVC名称+内部路径，或NFS连接地址+NFS内部路径）

**约束条件**:
- 纳管路径必须指向一个目录（而非单个文件），该目录包含模型权重的全部相关文件
- 纳管操作需记录操作日志

#### 3.2.2 纳管校验

**需求编号**: REQ-REGISTER-002

**需求描述**: 系统在纳管完成后，应自动对权重文件进行校验。

**校验内容**:
- 文件数量完整性校验：校验权重文件是否齐全（例如分片 safetensors 文件是否全部存在）
- 设计需保留扩展性，未来支持通过后台任务执行脚本检查权重文件完整性、SHA256 值与官方提供值是否匹配等

**触发时机**: 纳管完成后自动触发

### 3.3 权重上传

#### 3.3.1 上传任务创建

**需求编号**: REQ-UPLOAD-001

**需求描述**: 系统应支持创建权重上传任务，从外部存储将模型权重文件复制到平台统一管理的 PVC 中。

**上传定义**: 上传是指根据给定的外部存储路径（NFS/CIFS/PVC+路径），将权重文件从原路径拷贝到平台内置的 PVC 下，作为模型版本的权重来源。

**支持的源路径类型**:
- NFS 路径
- CIFS 路径
- PVC + 内部路径

**输入**: 源路径、目标模型 ID、上传配置

**约束条件**:
- 上传任务创建后自动开始执行
- 上传任务为异步后台任务
- 上传大文件时需支持断点续传
- 上传前需进行文件后缀白名单校验（见 3.6.1）
- 上传操作需记录操作日志

#### 3.3.2 上传任务管理

**需求编号**: REQ-UPLOAD-002

**需求描述**: 系统应支持对上传任务进行管理。

**支持的操作**:
- 查看：查看上传任务的执行状态、进度、日志等信息
- 停止（暂停）：暂停正在执行的上传任务
- 恢复：恢复已暂停的上传任务
- 删除：删除上传任务记录

**约束条件**:
- 删除任务时，如文件已部分上传，系统应清理已上传的部分文件，避免残留
- 仅任务创建者或具有相应操作权限的用户可管理任务

### 3.4 信息展示

#### 3.4.1 权重类型识别

**需求编号**: REQ-INFO-001

**需求描述**: 系统应支持通过解析权重目录中的 config.json 文件，识别模型权重的数据类型并展示。

**识别内容**: 包括但不限于 FP16、w8a8 等数据类型

**触发时机**:
- 权重导入（纳管/上传）完成后自动触发
- 支持定期自动重新识别
- 支持用户手动触发重新识别

**约束条件**:
- 如 config.json 不存在或格式异常，需标记识别失败状态

#### 3.4.2 权重文件完整性校验

**需求编号**: REQ-INFO-002

**需求描述**: 系统应支持对权重文件进行完整性校验。

**当前校验内容**:
- 文件数量完整性：校验权重文件是否齐全（例如分片 safetensors 文件是否全部存在）

**扩展性要求**:
- 设计需保留扩展性，未来支持通过后台任务执行脚本检查权重文件完整性、SHA256 值与官方提供值是否匹配等

**触发时机**:
- 权重导入（纳管/上传）完成后自动触发
- 支持定期自动重新校验
- 支持用户手动触发重新校验

**约束条件**:
- 校验结果需持久化并展示给用户
- 校验失败需标记版本状态为异常

### 3.5 模型分类管理

#### 3.5.1 模型分类管理

**需求编号**: REQ-CATEGORY-001

**需求描述**: 系统应支持模型分类的管理。

**分类层级**: 两层结构
- 第一层：模型分类（Classification）
- 第二层：模型类型（Type），从属于某一模型分类

**示例**:
- TextGeneration（分类）→ glm-5（类型）
- ImageTextToText（分类）→ Qwen2.5-VL-7B（类型）

**支持的操作**:
- 查看分类列表
- 添加自定义分类/类型
- 删除分类/类型

**约束条件**:
- 系统提供预设的分类和类型
- 用户可添加自定义分类和类型
- 删除分类/类型时，如该分类/类型下仍有模型，禁止删除
- 每个模型权重必须归属于一个模型分类及一个模型类型

#### 3.5.2 模型标签管理

**需求编号**: REQ-TAG-001

**需求描述**: 系统应支持为模型添加自定义标签（Tag）。

**支持的操作**:
- 为模型添加标签
- 为模型移除标签
- 通过标签筛选模型

**约束条件**:
- 标签为字符串类型
- 同一模型可拥有多个标签
- 标签不强制唯一，不同模型可使用相同标签

#### 3.5.3 模型分类/类型修改

**需求编号**: REQ-CATEGORY-002

**需求描述**: 系统应支持修改已有模型的分类和类型。

**约束条件**:
- 修改后需保证新分类+类型组合下模型名称唯一性约束不被违反
- 仅模型创建者或具有相应操作权限的用户可修改

### 3.6 安全与校验

#### 3.6.1 文件后缀白名单

**需求编号**: REQ-SECURITY-001

**需求描述**: 系统应维护一个权重文件后缀白名单，在权重上传前进行校验。

**白名单管理方式**:
- 全局统一白名单
- 白名单通过 ConfigMap 挂载到服务中，不存储于数据库
- 保留在后台直接修改 ConfigMap 的逃生通道

**校验行为**:
- 上传前校验文件后缀是否在白名单中
- 如文件后缀不在白名单中，拒绝上传并提示用户
- 如文件存在安全风险（即使后缀在白名单中），提示用户安全问题

#### 3.6.2 SSL/TLS 双向证书认证

**需求编号**: REQ-SECURITY-002

**需求描述**: 系统应支持 SSL/TLS 双向证书认证，确保通信安全。

**适用范围**: 所有对外暴露的接口（人机接口和机机接口）

### 3.7 分权分域

#### 3.7.1 资源组可见性控制

**需求编号**: REQ-RBAC-001

**需求描述**: 系统应基于资源组对模型的可见性进行控制。

**资源组类型**:
- 普通资源组：组内用户只能访问本资源组的资源和 public 资源组的资源
- admin_group：仅包含 admin 账户，admin 可访问所有资源组的资源
- public 资源组：组内资源可被所有账户访问

**可见性规则**:
- 普通用户只能看到本资源组和 public 资源组的模型
- admin 可看到所有资源组的模型
- 资源组控制的是「可见性」（能看到哪些模型），操作权限（CRUD）复用平台已有的 RBAC 鉴权体系

#### 3.7.2 模型资源组归属

**需求编号**: REQ-RBAC-002

**需求描述**: 每个模型必须归属于一个资源组。

**约束条件**:
- 模型创建时指定资源组
- 当前不支持修改模型的资源组
- 设计上需保留后续支持修改模型资源组的可扩展性

#### 3.7.3 public 资源组权限

**需求编号**: REQ-RBAC-003

**需求描述**: public 资源组中的模型权限规则。

**权限规则**:
- 任何用户都可以在 public 资源组下创建模型
- 模型的创建者可修改/删除自己在 public 资源组下创建的模型
- admin 可管理 public 资源组下的所有模型

### 3.8 对接其他模块

#### 3.8.1 机机接口 — 查询权重路径

**需求编号**: REQ-M2M-001

**需求描述**: 系统应提供机机接口，供其他模块查询模型权重版本的 PVC 路径及内部路径，供训推任务的 Pod 挂载使用。

**输入**: 模型 ID、版本号

**输出**: PVC 名称、内部路径

**约束条件**:
- 机机接口不经过 Gateway，由调用方直接访问模型仓库 Pod 的 URL
- 机机接口通过 SSL 证书校验进行认证
- 调用方需具有访问该模型版本的权限

#### 3.8.2 机机接口 — 训练权重归档

**需求编号**: REQ-M2M-002

**需求描述**: 系统应提供机机接口，支持训练任务将产生的新权重归档为新版本。

**流程**:
1. 训练模块调用模型仓库机机接口，请求创建新版本
2. 模型仓库创建新版本，分配存储路径
3. 训练模块另起任务，将新生成的权重文件移动至模型仓库分配的路径下
4. 模型仓库在检测到权重文件到位后，更新版本状态并执行校验

**约束条件**:
- 机机接口通过 SSL 证书校验进行认证
- 归档操作需记录操作日志

### 3.9 操作日志

#### 3.9.1 操作日志记录

**需求编号**: REQ-LOG-001

**需求描述**: 系统应记录模型仓库模块的关键操作日志，并上报到平台统一日志服务。

**需记录的操作**:
- 模型创建、修改、删除（软删除/硬删除）、恢复
- 版本创建、删除（软删除/硬删除）、恢复
- 版本锁定、解锁
- 上传任务创建、停止、恢复、删除
- 转换任务创建、删除
- 分类/类型添加、删除
- 标签添加、移除

**日志内容**:
- 操作者身份
- 操作类型
- 操作对象（模型 ID、版本号等）
- 操作时间
- 操作结果（成功/失败）
- 失败原因（如适用）

### 3.10 权重格式转换

#### 3.10.1 权重格式转换

**需求编号**: REQ-CONVERT-001

**需求描述**: 系统应支持模型权重格式转换功能，将权重从一种格式转换为另一种格式，并保存为新版本。

**支持的转换**:
- Megatron 格式 → Safetensors 格式

**流程**:
1. 用户选择源模型版本和转换目标格式
2. 系统创建转换任务，异步执行格式转换
3. 转换完成后，系统创建新版本保存转换后的权重
4. 新版本继承源模型的元数据（分类、类型、标签等），版本号按自动递增规则分配

**约束条件**:
- 转换任务为异步后台任务
- 转换过程中源版本不能被删除（自动锁定源版本）
- 转换完成后自动解锁源版本
- 转换失败需标记任务状态并记录失败原因
- 转换操作需记录操作日志
- 仅具有相应操作权限的用户可执行转换

#### 3.10.2 转换任务管理

**需求编号**: REQ-CONVERT-002

**需求描述**: 系统应支持对权重格式转换任务进行管理。

**支持的操作**:
- 查看：查看转换任务的执行状态、进度、日志等信息
- 删除：删除转换任务记录

**约束条件**:
- 仅任务创建者或具有相应操作权限的用户可管理任务

### 3.11 通用功能

#### 3.11.1 日志轮转

**需求编号**: REQ-GENERAL-001

**需求描述**: 系统应支持使用 Log4j 进行日志轮转，确保日志文件不会无限增长。

#### 3.11.2 版本状态管理

**需求编号**: REQ-GENERAL-002

**需求描述**: 系统应支持模型版本的状态管理。

**版本状态**:
- 导入中（Importing）：权重文件正在导入（上传/复制中）
- 可用（Available）：权重文件已就绪，校验通过
- 导入失败（ImportFailed）：权重文件导入失败
- 校验失败（ValidationFailed）：权重文件完整性校验或类型识别失败
- 异常（Error）：其他异常状态

**状态转换规则**:
- 创建版本后初始状态为「导入中」
- 导入完成后进入「可用」或「导入失败」
- 校验完成后可能进入「校验失败」
- 用户可手动触发重新校验，状态相应更新

## 5. 接口需求

### 5.1 接口分类

模型仓库模块提供两类接口：

| 接口类型 | 使用者 | 路由方式 | 认证方式 |
|----------|--------|----------|----------|
| 人机接口（User API） | 前端页面、用户 | 通过 Gateway 路由 | 复用 Gateway 认证机制 |
| 机机接口（M2M API） | 平台其他模块（推理、训练等） | 直接调用 Pod URL | SSL 证书校验 |

### 5.2 人机接口需求

人机接口供前端页面调用，需支持以下功能：

- 模型创建、查看、修改、删除（软删除/硬删除）、恢复
- 版本创建、查看、删除（软删除/硬删除）、恢复
- 模型列表查询（分页、筛选、排序、搜索）
- 回收站管理（查看、恢复、硬删除）
- 上传任务创建、查看、停止、恢复、删除
- 模型分类/类型管理（查看、添加、删除）
- 标签管理（添加、移除）
- 权重类型识别触发、权重完整性校验触发
- 权重格式转换任务创建、查看

### 5.3 机机接口需求

机机接口供平台其他模块调用，需支持以下功能：

- 查询模型权重版本的 PVC 路径及内部路径（供训推任务 Pod 挂载）
- 查询模型版本是否为纳管类型（返回是否只读挂载）
- 训练任务新权重归档（创建新版本并分配路径）
- 权重版本锁定/解锁（供训推任务占用/释放权重）

#### 5.3.1 机机接口 — 权重版本锁定/解锁

**需求编号**: REQ-M2M-003

**需求描述**: 系统应提供机机接口，供训推任务对模型权重版本进行锁定和解锁操作。

**锁定功能**:
- 训推任务在开始使用某模型版本前，调用锁定接口
- 系统标记该版本为「已锁定」状态
- 锁定期间，该版本禁止被删除（软删除/硬删除）

**解锁功能**:
- 训推任务结束后，调用解锁接口
- 系统移除该版本的「已锁定」状态
- 解锁后，该版本恢复正常，可被删除

**约束条件**:
- 机机接口不经过 Gateway，由调用方直接访问模型仓库 Pod 的 URL
- 机机接口通过 SSL 证书校验进行认证
- 锁定操作需记录操作日志（记录锁定者、锁定时间）
- 解锁操作需记录操作日志（记录解锁者、解锁时间）
- 同一版本支持被多个任务同时锁定（引用计数或锁列表）
- 仅当版本的所有锁定都被解除后，才允许删除

| 实体 | 描述 |

| 约束 | 描述 |

| 资源组可见性 | 模型的可见性受资源组控制 |
