## 1. 战略设计概述

### 1.1 设计目标

本文档基于 DDD（领域驱动设计）方法论，对 ModelLite 模型仓库模块进行战略层面的设计。战略设计的核心目标是：

1. **识别核心域（Core Domain）**：找到模型仓库中最具业务价值、最需要深度建模的部分
2. **划分子域（Subdomains）**：将复杂业务拆分为可管理的子域
3. **定义限界上下文（Bounded Contexts）**：建立明确的模型边界，避免概念混淆
4. **构建上下文映射（Context Map）**：定义上下文间的集成关系和协作模式
5. **制定领域愿景声明（Domain Vision Statement）**：明确核心域的价值主张

### 1.3 设计原则

**原则一：业务优先于技术**
- 子域划分基于业务能力，而非技术实现
- 限界上下文边界反映业务语义边界

**原则二：深度建模核心域**
- 核心域投入最优质的建模资源
- 通用域采用成熟方案，避免过度设计

**原则三：显式边界**
- 每个限界上下文有明确的职责范围
- 跨上下文通信必须通过显式的集成模式

**原则四：语言一致性**
- 每个限界上下文内部使用统一的 ubiquitous language
- 同一术语在不同上下文中可以有不同的含义

## 2. 领域愿景声明（Domain Vision Statement）

### 2.1 核心域价值主张

> **ModelLite 模型仓库的核心价值在于：为机器学习模型提供全生命周期的版本化管理能力，确保模型权重资产的安全存储、高效流转和可靠追溯，使平台各模块能够便捷、安全地获取和使用模型资源。**

### 2.2 核心竞争力

模型仓库的核心竞争力体现在以下三个方面：

| 竞争力维度 | 描述 | 业务价值 |
|-----------|------|----------|
| **版本化管理** | 自动递增的版本号、完整的版本血缘追溯 | 支持模型迭代演进，确保训推任务使用正确的模型版本 |
| **安全管控** | 版本锁机制 + 资源组隔离 + 完整的操作审计 | 防止模型被误删，保障多租户环境下的数据安全 |
| **灵活接入** | 支持纳管、上传、归档、转换多种权重来源 | 适应不同的模型获取场景，降低用户使用门槛 |

### 2.3 战略投资方向

基于核心竞争力，战略投资应聚焦于：

1. **模型版本管理域**：最深度的建模，最严格的业务规则
2. **权重生命周期域**：复杂的状态机管理，可靠的异步任务执行
3. **版本锁与权限域**：精细化的并发控制和安全策略

## 3. 子域划分（Subdomain Classification）

### 3.1 子域识别方法

采用**业务事件风暴**方法，从需求规格说明书中提取核心业务能力和支撑能力：

1. 梳理所有业务用例和业务流程
2. 识别独立的业务概念和责任边界
3. 评估业务复杂度和差异化价值
4. 归类为核心域、支撑域、通用域

### 3.3 子域详细说明

#### 3.3.1 核心域 - 模型版本管理域（Model Version Management）

**子域定位**：模型仓库最核心的业务能力，直接体现平台的核心竞争力。

**业务范围**：
- 模型的创建、查询、修改、删除（软删除/硬删除）
- 模型版本的创建、查询、删除（软删除/硬删除）
- 版本号的自动递增分配
- 模型与版本的关联关系维护
- 模型元数据管理（名称、描述、分类、类型、标签等）

**业务复杂度**：★★★★★
- 复杂的唯一性约束（分类+类型+名称组合唯一）
- 级联删除逻辑（删除模型时级联删除版本）
- 版本号连续性约束
- 资源组可见性控制

**差异化价值**：高
- 版本化管理是模型仓库区别于简单文件存储的核心能力
- 直接影响下游训推任务的可靠性

**建模深度**：最深
- 需要完整的聚合设计、实体关系、业务规则
- 需要精细化的领域服务设计

**业务规则清单**：
| 规则编号 | 规则描述 | 优先级 |
|---------|----------|--------|
| R-MVM-001 | 同一分类+类型组合下，模型名称必须唯一 | 高 |
| R-MVM-002 | 模型创建时必须同时创建第一个版本 | 高 |
| R-MVM-003 | 版本号采用自动递增整数，不允许跳号 | 高 |
| R-MVM-004 | 模型名称创建后不可修改 | 高 |
| R-MVM-005 | 模型资源组创建后不可修改（设计保留扩展性） | 高 |
| R-MVM-006 | 修改分类/类型后需保证新组合下名称唯一性 | 高 |
| R-MVM-007 | 删除模型前需检查所有版本是否被锁定 | 高 |
| R-MVM-008 | 单模型版本数量上限为 50 个 | 高 |
| R-MVM-009 | 单资源组模型数量上限为 100 个 | 高 |

#### 3.3.2 核心域 - 权重生命周期管理域（Weight Lifecycle Management）

**子域定位**：管理模型权重文件从引入到使用的完整生命周期。

**业务范围**：
- 权重纳管（Register）：登记外部已有路径
- 权重上传（Upload）：从外部存储复制到平台 PVC
- 权重归档（Archive）：训练任务产出权重保存为新版本
- 权重转换（Convert）：格式转换（Megatron → Safetensors）
- 权重校验（Validation）：完整性校验、类型识别
- 版本状态管理：导入中 → 可用/导入失败/校验失败/异常

**业务复杂度**：★★★★★
- 多种权重来源方式，每种有不同的处理流程
- 复杂的异步任务状态机
- 上传任务支持暂停、恢复、断点续传
- 转换任务需要自动锁定/解锁源版本

**差异化价值**：高
- 灵活的权重接入能力是平台易用性的关键
- 可靠的异步任务执行保障数据完整性

**建模深度**：深
- 需要设计 UploadTask、ConvertTask 等聚合
- 需要设计 WeightValidationService 等领域服务
- 需要处理异步任务的可靠执行

**状态机设计**：
```
                    ┌─────────────────────────────────────────┐
                    │                                         │
                    ▼                                         │
NO_WEIGHT ─────────► UPLOADING ─────────► AVAILABLE           │
       │                  │                                  │
       │                  ├──► IMPORT_FAILED                  │
       │                  │         │                       │
       │                  │         └──► (可重新上传)         │
       │                  │                                │
       │                  ├──► VALIDATION_FAILED            │
       │                  │         │                       │
       │                  │         └──► (可重新校验)         │
       │                  │                                │
       │                  └──► ERROR                        │
       │                                                   │
       └───────────────────────────────────────────────────┘
       (纳管/归档/转换创建版本后直接进入校验流程)
```

**业务规则清单**：
| 规则编号 | 规则描述 | 优先级 |
|---------|----------|--------|
| R-WLM-001 | 纳管路径必须指向目录而非单个文件 | 高 |
| R-WLM-002 | 纳管版本只能只读挂载 | 高 |
| R-WLM-003 | 上传前需校验文件后缀白名单 | 高 |
| R-WLM-004 | 上传任务支持断点续传 | 高 |
| R-WLM-005 | 删除上传任务时需清理已上传的部分文件 | 高 |
| R-WLM-006 | 转换过程中源版本自动锁定，完成后自动解锁 | 高 |
| R-WLM-007 | 权重导入完成后自动触发校验和类型识别 | 高 |
| R-WLM-008 | 校验失败标记版本为异常状态 | 中 |

#### 3.3.3 核心域 - 版本锁与权限管控域（Version Lock & Access Control）

**子域定位**：保障模型版本在使用期间不被误删，实现精细化的并发控制。

**业务范围**：
- 版本锁定：训推任务占用版本时加锁
- 版本解锁：训推任务结束后释放锁
- 锁续约：长时间任务延长锁有效期
- 锁过期清理：Leader 节点巡检清理僵尸锁
- 资源组可见性控制
- RBAC 权限集成

**业务复杂度**：★★★★☆
- 支持多任务同时锁定同一版本（引用计数）
- TTL 自动过期 + 手动续约机制
- Leader 选举保证清理任务单点执行
- 资源组多层级可见性规则

**差异化价值**：高
- 版本锁是保障训推任务稳定运行的关键机制
- 资源组隔离是多租户安全的基础

**建模深度**：深
- 需要设计 VersionLock 实体和锁管理领域服务
- 需要处理分布式环境下的锁一致性

**业务规则清单**：
| 规则编号 | 规则描述 | 优先级 |
|---------|----------|--------|
| R-VLAC-001 | 同一版本支持被多个任务同时锁定 | 高 |
| R-VLAC-002 | 仅当版本的所有锁定都解除后才允许删除 | 高 |
| R-VLAC-003 | 锁默认 TTL 为 24 小时 | 高 |
| R-VLAC-004 | 锁支持续约，每次续约重置 TTL | 高 |
| R-VLAC-005 | Leader 节点每 10 分钟巡检清理过期锁 | 高 |
| R-VLAC-006 | 普通用户只能看到本资源组和 public 资源组的模型 | 高 |
| R-VLAC-007 | admin 可看到所有资源组的模型 | 高 |
| R-VLAC-008 | public 资源组中任何用户都可创建模型 | 高 |

#### 3.3.4 支撑域 - 分类体系域（Classification System）

**子域定位**：为模型提供分类组织体系，便于模型检索和管理。

**业务范围**：
- 模型分类（Classification）管理：一级分类
- 模型类型（ModelType）管理：二级分类
- 分类与类型的层级关系维护
- 内置分类/类型与用户自定义分类/类型的区分

**业务复杂度**：★★☆☆☆
- 两层层级结构
- 删除约束（下有模型时禁止删除）

**差异化价值**：中
- 分类体系是模型组织的基础，但非核心竞争力
- 可通过 UI 设计弥补分类体系的不足

**建模深度**：中等
- 需要设计 Category 和 ModelType 实体
- 需要处理分类删除时的引用检查

**业务规则清单**：
| 规则编号 | 规则描述 | 优先级 |
|---------|----------|--------|
| R-CS-001 | 系统提供预设分类和类型 | 高 |
| R-CS-002 | 用户可添加自定义分类和类型 | 中 |
| R-CS-003 | 删除分类/类型时，如下有模型则禁止删除 | 高 |
| R-CS-004 | 每个模型必须归属于一个分类和一个类型 | 高 |

---

#### 3.3.5 支撑域 - 标签管理域（Tag Management）

**子域定位**：为模型提供灵活的标签组织能力，支持多维度筛选。

**业务范围**：
- 标签的创建、删除
- 为模型添加/移除标签
- 通过标签筛选模型

**业务复杂度**：★★☆☆☆
- 简单的多对多关联
- 标签不强制唯一

**差异化价值**：低
- 标签是通用的组织方式，非差异化能力

**建模深度**：浅
- Tag 实体 + ModelTag 关联实体

**业务规则清单**：
| 规则编号 | 规则描述 | 优先级 |
|---------|----------|--------|
| R-TM-001 | 同一模型最多 20 个标签 | 中 |
| R-TM-002 | 单个标签长度 1-50 字符 | 中 |
| R-TM-003 | 标签不强制唯一 | 低 |

---

#### 3.3.6 支撑域 - 回收站管理域（Recycle Bin Management）

**子域定位**：管理被软删除的模型和版本，提供恢复和彻底删除能力。

**业务范围**：
- 回收站列表查询
- 从回收站恢复模型/版本
- 从回收站硬删除模型/版本
- 回收站可见性控制

**业务复杂度**：★★★☆☆
- 全局统一列表但受资源组过滤
- 恢复操作需还原软删除状态

**差异化价值**：中
- 回收站是数据安全的重要保障，但实现方式相对标准

**建模深度**：中等
- 复用 Model 和 ModelVersion 的软删除字段
- 需要设计 RecycleBinQueryService

**业务规则清单**：
| 规则编号 | 规则描述 | 优先级 |
|---------|----------|--------|
| R-RB-001 | 回收站为全局统一列表 | 高 |
| R-RB-002 | 回收站可见性受资源组控制 | 高 |
| R-RB-003 | 恢复后模型/版本恢复到软删除前状态 | 高 |
| R-RB-004 | 硬删除时同时删除存储上的实际权重文件 | 高 |

#### 3.3.7 通用域 - 操作审计域（Audit Logging）

**子域定位**：记录所有关键操作，支持审计追溯。

**业务范围**：
- 操作日志记录
- 日志格式标准化
- 日志上报到平台统一日志服务

**业务复杂度**：★★☆☆☆
- 标准化的日志格式
- 异步上报机制

**差异化价值**：低
- 审计是合规要求，非业务差异化能力
- 复用平台统一日志服务

**建模深度**：浅
- 使用 Annotation + AOP 实现
- 无需复杂的领域模型

---

#### 3.3.8 通用域 - 安全认证域（Security & Authentication）

**子域定位**：保障接口通信安全和身份认证。

**业务范围**：
- SSL/TLS 双向证书认证
- User API 的 Gateway Header 认证透传
- M2M API 的证书校验
- 文件后缀白名单校验

**业务复杂度**：★★★☆☆
- SSL 双向认证配置
- 白名单动态加载（ConfigMap）

**差异化价值**：低
- 安全是基础设施要求，非业务差异化能力
- 复用平台 RBAC 体系

**建模深度**：浅
- 使用 Spring Security + SSL 配置
- 白名单通过 ConfigMap 外部化配置

---

#### 3.3.9 通用域 - 任务调度域（Task Scheduling）

**子域定位**：提供异步任务的可靠执行能力。

**业务范围**：
- K8s Job 的创建和管理
- fabric8 Informer 状态监听
- 任务状态同步
- Leader Election 机制

**业务复杂度**：★★★★☆
- K8s Job 生命周期管理
- 多副本环境下的 Leader 选举
- 任务状态可靠同步

**差异化价值**：低
- 任务调度是基础设施能力
- 复用 K8s 原生能力

**建模深度**：中等
- 需要封装 K8s API 调用
- 需要处理 Job 状态监听和同步

## 4. 限界上下文划分（Bounded Contexts）

### 4.1 划分原则

基于子域划分和业务能力聚合，遵循以下原则定义限界上下文：

1. **语义一致性原则**：上下文内的术语具有唯一、明确的含义

3. **独立演化原则**：上下文可以独立开发、部署、演进

### 5.2 上下文间集成关系

#### 5.2.1 模型管理上下文 ↔ 权重生命周期上下文

**关系类型**：Customer-Supplier（客户-供应商）

**集成方向**：双向

**集成说明**：
- 模型管理上下文是权重生命周期上下文的客户：创建版本后需要触发权重导入流程
- 权重生命周期上下文是模型管理上下文的客户：上传/转换完成后需要更新版本状态

**集成模式**：
- 模型管理上下文通过领域事件 `VersionCreatedEvent` 通知权重生命周期上下文
- 权重生命周期上下文通过领域事件 `VersionStatusChangedEvent` 通知模型管理上下文
- 使用异步消息机制解耦

**Anti-Corruption Layer**：
- 权重生命周期上下文提供 `WeightImportRequest` 值对象作为接口契约
- 模型管理上下文不直接依赖权重生命周期上下文的内部模型

---

#### 5.2.2 模型管理上下文 → 分类管理上下文

**关系类型**：Customer-Supplier（客户-供应商）

**集成方向**：单向（模型管理 → 分类管理）

**集成说明**：
- 模型管理上下文需要查询分类和类型信息来创建和展示模型
- 分类管理上下文独立演化，不依赖模型管理上下文

**集成模式**：
- 模型管理上下文调用分类管理上下文的查询接口
- 使用 `CategoryId`、`TypeId` 值对象作为引用标识

**Anti-Corruption Layer**：
- 模型管理上下文内部使用 `CategoryReference`、`TypeReference` 值对象
- 不直接使用分类管理上下文的 Category 聚合

---

#### 5.2.3 模型管理上下文 → 版本锁与权限控制上下文

**关系类型**：Customer-Supplier（客户-供应商）

**集成方向**：单向（模型管理 → 版本锁与权限控制）

**集成说明**：
- 模型管理上下文在删除模型/版本前，需要检查是否被锁定
- 模型管理上下文在查询模型列表时，需要应用资源组可见性过滤

**集成模式**：
- 调用 `LockManagementService.isVersionLocked(versionId)` 方法
- 调用 `ResourceGroupService.getVisibleResourceGroups(userId)` 方法

**Anti-Corruption Layer**：
- 模型管理上下文使用 `LockCheckResult` 值对象接收检查结果
- 不直接依赖版本锁的内部实现

---

#### 5.2.4 权重生命周期上下文 → 版本锁与权限控制上下文

**关系类型**：Customer-Supplier（客户-供应商）

**集成方向**：单向（权重生命周期 → 版本锁与权限控制）

**集成说明**：
- 权重转换任务需要锁定源版本，防止转换过程中被删除
- 转换完成后需要解锁源版本

**集成模式**：
- 调用 `LockManagementService.lockVersion()` 和 `unlockVersion()` 方法
- 使用领域事件 `VersionLockedEvent`、`VersionUnlockedEvent` 通知

#### 5.2.5 权重生命周期上下文 → 任务调度上下文

**关系类型**：Customer-Supplier（客户-供应商）

**集成方向**：单向（权重生命周期 → 任务调度）

**集成说明**：
- 权重生命周期上下文需要任务调度上下文执行实际的 K8s Job
- 任务调度上下文提供通用的任务执行能力

**集成模式**：
- 权重生命周期上下文创建 `JobSpec` 值对象，提交给任务调度上下文
- 任务调度上下文通过回调或事件通知任务状态变化

**Anti-Corruption Layer**：
- 任务调度上下文提供 `TaskExecutionGateway` 接口
- 权重生命周期上下文不直接依赖 K8s API

**注意**：

- 安全认证策略属于审计与安全上下文的内部实现

## 7. 战略设计决策记录

### 7.1 关键架构决策

#### 决策 1：模型与版本的关系设计

**决策内容**：Model 作为聚合根，ModelVersion 作为其内部实体

**决策理由**：
- 模型和版本在业务上强关联，版本不能独立存在（必须属于某个模型）
- 删除模型时需要级联删除所有版本，这是聚合内的一致性边界
- 版本号的分配需要模型级别的控制（保证连续性）

**替代方案**：ModelVersion 作为独立聚合根
-  rejected 原因：版本号连续性约束需要在模型级别控制，如果版本是独立聚合，需要通过领域服务协调，增加复杂度

#### 决策 2：版本锁的设计位置

**决策内容**：VersionLock 作为实体（非聚合根），依附于 ModelVersion

**决策理由**：
- 版本锁的生命周期完全依附于版本，不能独立存在
- 锁的创建和释放由版本的使用方（训推任务）触发，但锁的存在意义在于保护版本
- 将锁放在模型管理上下文中，便于删除版本时统一检查

**替代方案**：版本锁作为独立聚合根
-  rejected 原因：锁没有独立的业务意义，且需要频繁与版本关联查询

#### 决策 3：上传任务和转换任务的聚合边界

**决策内容**：UploadTask 和 ConvertTask 作为独立的聚合根

**决策理由**：
- 上传任务和转换任务有独立的生命周期，不依附于模型或版本
- 任务可以独立创建、执行、暂停、删除
- 任务状态的变化需要独立追踪和持久化

**替代方案**：任务作为 ModelVersion 的内部实体
-  rejected 原因：任务的生命周期独立于版本，且需要支持独立查询和管理

#### 决策 4：分类管理的独立上下文

**决策内容**：分类管理作为独立的限界上下文

**决策理由**：
- 分类体系是相对稳定的参考数据，与模型管理有不同的变更频率
- 分类管理可以被其他模块复用（如模型市场、模型推荐等）
- 独立的上下文允许分类体系独立演化和扩展

**替代方案**：分类管理作为模型管理上下文的一部分
-  rejected 原因：增加模型管理上下文的复杂度，且不利于未来复用

#### 决策 5：审计与安全的共享内核

**决策内容**：审计日志格式作为 Shared Kernel，安全认证作为独立上下文

**决策理由**：
- 审计日志格式需要所有上下文统一，便于日志收集和分析
- 安全认证实现复杂，且需要统一配置，适合作为独立上下文
- 共享内核应保持最小化，避免过度耦合

### 7.2 与原有架构设计的差异

| 方面 | 原有架构设计 (v1.1) | DDD 战略设计 | 差异说明 |
|------|-------------------|-------------|---------|
| 模块划分 | 按技术层划分（接口层、服务层、仓储层、基础设施层） | 按业务能力划分（限界上下文） | 从分层架构转向领域架构 |
| 服务设计 | 按实体设计服务（ModelService、VersionService...） | 按业务用例设计领域服务 | 服务粒度更贴近业务 |
| 聚合边界 | 未明确聚合概念 | 明确 4 个聚合根 | 引入聚合一致性边界 |
| 上下文边界 | 未定义 | 定义 6 个限界上下文 | 建立显式模型边界 |
| 跨模块通信 | 服务直接调用 | 领域事件 + ACL | 降低耦合，提高内聚 |
| 业务规则 | 散落在服务代码中 | 集中在聚合和领域服务中 | 业务规则显性化 |

## 8. 实施路线图

### 8.1 阶段划分

基于战略设计，建议按以下阶段实施：

#### 第一阶段：核心域基础（P0）

**目标**：建立模型版本管理和权重生命周期管理的基础能力

**范围**：
- 模型管理上下文：模型 CRUD、版本 CRUD、回收站
- 权重生命周期上下文：纳管、上传、归档、状态管理
- 版本锁与权限控制上下文：锁定、解锁、资源组可见性

**预期产出**：
- 可运行的模型仓库基础版本
- 支持模型的创建、上传、查询、删除

#### 第二阶段：支撑域完善（P1）

**目标**：完善分类体系、回收站、任务调度能力

**范围**：
- 分类管理上下文：分类/类型 CRUD
- 回收站管理：完整的回收站功能
- 任务调度上下文：可靠的异步任务执行

**预期产出**：
- 完整的模型组织体系
- 可靠的异步任务执行

#### 第三阶段：通用域集成（P2）

**目标**：集成审计日志、安全认证、标签管理

**范围**：
- 审计与安全上下文：操作日志、SSL 认证、白名单
- 标签管理：标签 CRUD、模型标签关联

**预期产出**：
- 完整的安全和审计能力
- 灵活的标签组织

#### 第四阶段：高级特性（P3）

**目标**：支持权重转换、高级查询、性能优化

**范围**：
- 权重转换：Megatron → Safetensors
- 高级查询：全文搜索、复杂筛选
- 性能优化：缓存、索引优化

## 9. 风险评估与缓解策略

### 9.1 设计风险

| 风险 | 影响 | 可能性 | 缓解策略 |
|------|------|--------|---------|
| 聚合边界划分不当 | 高 | 中 | 通过事件风暴验证聚合边界；保持聚合小且聚焦 |
| 上下文间通信过度复杂 | 中 | 中 | 优先使用领域事件异步通信；减少同步调用 |
| 版本锁性能瓶颈 | 高 | 中 | 使用数据库索引优化；考虑缓存锁定状态 |
| 异步任务可靠性 | 高 | 中 | 设计任务状态机；实现任务恢复机制 |

### 9.2 实施风险

| 风险 | 影响 | 可能性 | 缓解策略 |
|------|------|--------|---------|
| 团队 DDD 经验不足 | 中 | 高 | 提供 DDD 培训；从核心域开始实践 |
| 与旧系统兼容 | 高 | 高 | 设计防腐层；逐步迁移 |
| 性能不达标 | 高 | 中 | 早期进行性能测试；预留优化空间 |
